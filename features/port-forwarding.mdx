Kubewall's one-click port forwarding feature allows you to securely access applications running in your Kubernetes cluster without creating external LoadBalancers or Ingress resources.

## Overview

Port forwarding creates a secure tunnel between your local machine and a pod or service in your cluster, enabling local access to remote applications.

## Starting Port Forwarding

### From Pod Details

1. Navigate to any pod detail view
2. Click the **Port Forward** icon (plug icon) in the header
3. Configure the port forwarding in the dialog
4. Click **Submit** to start forwarding

### From Service Details

1. Open any service detail view
2. Click the **Port Forward** icon
3. Select the service port to forward
4. Configure the local port
5. Click **Submit** to establish the connection

<Note>
The port forward icon shows a different state when active forwarding exists for the resource.
</Note>

## Configuration Dialog

The port forwarding dialog provides several configuration options:

<Tabs>
  <Tab title="Local Port">
    **Local Port**: The port on your machine that will receive traffic

    - Enter any available port number (1-65535)
    - Set to `0` for automatic assignment
    - Common ports: 8080, 3000, 5000, etc.
    - Kubewall validates port availability

    <Accordion title="Automatic Port Assignment">
    Setting the local port to `0` tells Kubernetes to automatically assign an available port:

    - Useful when you don't have a preference
    - Avoids port conflicts
    - Kubewall displays the assigned port after creation
    - Recommended for temporary debugging sessions
    </Accordion>
  </Tab>
  <Tab title="Container/Service Port">
    **Container Port (Pods)**: Select which container and port to forward to

    - Dropdown shows all containers in the pod
    - Lists all exposed ports for each container
    - Format: `container-name: port-number`
    - Can select custom port if not in dropdown

    **Service Port (Services)**: Select which service port to forward

    - Shows all defined service ports
    - Format: `port-name: port-number/protocol`
    - Automatically maps to target pod ports
  </Tab>
</Tabs>

## Managing Port Forwards

### Viewing Active Forwards

To see all active port forwarding sessions:

1. Navigate to the **Port Forwards** resource list
2. Or click the link in the port forwarding dialog
3. View all active forwards across your cluster

### Port Forward List Displays

- **Resource Name**: Pod or service being forwarded
- **Resource Type**: Pod or Service
- **Namespace**: Resource namespace
- **Local Port**: Your machine's port
- **Container/Service Port**: Target port in the cluster
- **Status**: Active or Error

<Note>
Port forwarding sessions are managed per cluster. Switching clusters shows different active forwards.
</Note>

### Stopping Port Forwards

Remove port forwards when no longer needed:

1. Go to the **Port Forwards** list
2. Select the forwards to stop (checkboxes)
3. Click **Delete** to stop them
4. Forwards are immediately terminated

<Accordion title="Automatic Cleanup">
Port forwards are automatically stopped when:
- The Kubewall backend restarts
- The target pod is deleted
- The network connection is lost
- The session exceeds timeout limits
</Accordion>

## Using Port Forwarded Applications

Once port forwarding is established:

### Web Applications

```bash
# In your browser
http://localhost:8080

# Or with curl
curl http://localhost:8080
```

### Database Connections

```bash
# PostgreSQL
psql -h localhost -p 5432 -U username -d database

# MySQL
mysql -h localhost -P 3306 -u username -p

# MongoDB
mongo localhost:27017/database
```

### Application Development

```javascript
// Connect your local app to forwarded service
const api = axios.create({
  baseURL: 'http://localhost:8080'
});
```

### API Testing

```bash
# Test API endpoints
curl http://localhost:8080/api/health

# With Postman
# Use localhost:8080 as base URL
```

## Multiple Port Forwards

You can create multiple port forwards simultaneously:

### Same Resource

Forward multiple ports from the same pod or service:

```
Pod: web-app
  8080 -> 80 (HTTP)
  8443 -> 443 (HTTPS)
  9090 -> 9090 (Metrics)
```

### Different Resources

Forward ports from multiple resources:

```
Frontend Pod: 3000 -> 3000
Backend Pod: 8080 -> 8080
Database Pod: 5432 -> 5432
Redis Pod: 6379 -> 6379
```

<Warning>
Each port forward consumes a connection to the Kubewall backend. Limit active forwards to only what you need.
</Warning>

### Port Management Tips

1. **Consistent Ports**: Use the same local ports for the same services
2. **Auto-Assignment**: Use port 0 when you don't need a specific port
3. **Avoid Conflicts**: Check for existing processes on local ports
4. **Multiple Clusters**: Use different port ranges per cluster

## Troubleshooting

<Accordion title="Port Already in Use">
**Error**: Local port is already bound

**Solution**:
- Choose a different local port
- Find and stop the process using the port
- Use automatic port assignment (port 0)
- Check for existing port forwards
</Accordion>

<Accordion title="Connection Refused">
**Error**: Cannot connect to localhost:port

**Solution**:
- Verify port forward is active in Port Forwards list
- Check pod is running and healthy
- Ensure target port is correct
- Review pod logs for binding issues
</Accordion>

<Accordion title="Slow Performance">
**Issue**: Forwarded connection is slow

**Causes**:
- Network latency to cluster
- Kubewall backend resource constraints
- Multiple active port forwards
- Large data transfers over tunnel

**Solution**:
- Reduce number of active forwards
- Use for debugging, not production traffic
- Consider direct cluster access for high throughput
</Accordion>

<Accordion title="Forward Disconnects">
**Issue**: Port forward stops unexpectedly

**Causes**:
- Pod restarted or deleted
- Kubewall backend restarted
- Network interruption
- Idle timeout

**Solution**:
- Check pod status
- Recreate the port forward
- Verify Kubewall backend is running
- Use port forward as temporary tool, not permanent solution
</Accordion>

## Security Considerations

<Warning>
Port forwards expose cluster services to your local machine. Be aware of security implications.
</Warning>

### Security Best Practices

- **Localhost Only**: Forwards bind to 127.0.0.1 by default
- **No External Access**: Other machines cannot access your forwards
- **RBAC Respected**: You can only forward to pods you have access to
- **No Encryption**: Traffic between Kubewall and cluster is encrypted, but localhost is not
- **Temporary Use**: Port forwards are for debugging, not production access

### When to Use Port Forwarding

✅ **Good Use Cases**:
- Temporary debugging access
- Local development testing
- Database administration
- API testing
- Metrics inspection

❌ **Avoid For**:
- Production traffic
- Long-running connections
- High-throughput data transfer
- Multiple users needing access
- Security-sensitive operations over untrusted networks
