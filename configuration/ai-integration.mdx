## AI Assistant Capabilities

The Kubewall AI assistant (kubewall-ai) can:

<Steps>
  <Step title="Analyze Kubernetes Resources">
    Examine manifests, identify issues, and suggest improvements for deployments, services, and other resources.
  </Step>

  <Step title="Troubleshoot Problems">
    Investigate pod failures, resource constraints, networking issues, and configuration errors.
  </Step>

  <Step title="Generate Manifests">
    Create complete, production-ready Kubernetes manifests based on your requirements.
  </Step>

  <Step title="Validate Configurations">
    Check for deprecated APIs, security risks, misconfigurations, and RBAC issues.
  </Step>

  <Step title="Link Related Resources">
    Connect and validate relationships between Deployments, Services, ConfigMaps, Secrets, and PVCs.
  </Step>

  <Step title="Optimize Resources">
    Recommend CPU/memory settings, storage configurations, and scaling strategies.
  </Step>
</Steps>

## Setting Up AI Integration

<Steps>
  <Step title="Navigate to AI Settings">
    Open Kubewall and go to the AI configuration section in the settings.
  </Step>

  <Step title="Add AI Provider">
    Click "Add Provider" and select your preferred AI service from the list.
  </Step>

  <Step title="Configure Credentials">
    Enter your API key and configure the base URL if needed:
    
    - **API Key**: Your provider's authentication key
    - **Base URL**: API endpoint (pre-filled with defaults)
    - **Model**: Specific model name to use
    - **Alias**: Friendly name for this configuration
  </Step>

  <Step title="Set as Default">
    Optionally mark this provider as your default for new conversations.
  </Step>

  <Step title="Test Connection">
    Start a conversation to verify the configuration works correctly.
  </Step>
</Steps>

## Provider Implementation

Kubewall uses the Vercel AI SDK to integrate with providers. Here's how each provider is initialized:

<CodeGroup>
```typescript client/src/components/app/kwAI/Chat/index.tsx (xAI)
case "xai":
  return createXai({
    apiKey: providerData.apiKey,
    baseURL: providerData.url,
    headers: {
      'HTTP-Referer': 'https://kubewall.com',
      'X-Title': 'Kubewall'
    }
  });
```

```typescript client/src/components/app/kwAI/Chat/index.tsx (Anthropic)
case "anthropic":
  return createAnthropic({
    apiKey: providerData.apiKey,
    baseURL: providerData.url,
    headers: {
      'HTTP-Referer': 'https://kubewall.com',
      'X-Title': 'Kubewall'
    }
  });
```

```typescript client/src/components/app/kwAI/Chat/index.tsx (Ollama)
case "ollama":
  return createOpenAICompatible({
    name: 'ollama',
    baseURL: `${providerData.url}/`,
    fetch: (url, options) => {
      const encodedUrl = encodeURIComponent(url.toString());
      const newUrl = `${API_VERSION}${MCP_SERVER_ENDPOINT}/${encodedUrl}?${clusterConfigKey}`;
      if (options) {
        options.headers = {
          ...options.headers,
          'HTTP-Referer': 'https://kubewall.com',
          'X-Title': 'Kubewall'
        };
      }
      return fetch(newUrl, options);
    },
    apiKey: providerData.apiKey,
    includeUsage: true
  });
```
</CodeGroup>

## Local Models with Ollama

For privacy-sensitive environments, use Ollama to run models locally:

<Steps>
  <Step title="Install Ollama">
    Download and install Ollama from [ollama.ai](https://ollama.ai)
  </Step>

  <Step title="Pull a model">
    ```bash
    ollama pull llama3.1
    ```
  </Step>

  <Step title="Verify Ollama is running">
    ```bash
    curl http://127.0.0.1:11434/v1/models
    ```
  </Step>

  <Step title="Configure in Kubewall">
    Add Ollama as a provider with URL `http://127.0.0.1:11434/v1`
  </Step>
</Steps>

<Note>
Local models require significant system resources. Ensure your machine has sufficient RAM and CPU for the model you choose.
</Note>

## AI Features

### Automatic Context Injection

When using AI on resource detail pages, Kubewall automatically includes the current resource's YAML in the conversation context.

### Reasoning Display

Kubewall displays the AI's reasoning process in a collapsible section, helping you understand how it arrived at its conclusions.

### Tool Calling

The AI can autonomously call Kubernetes API tools to:
- List resources across namespaces
- Get detailed resource information
- Analyze logs and events
- Check cluster state

### Token Usage Tracking

Each AI response shows:
- Prompt tokens (input)
- Completion tokens (output)  
- Total tokens used

## Best Practices

1. **Choose the right model**: Larger models (GPT-4, Claude 3 Opus) provide better analysis but cost more
2. **Use local models for sensitive data**: Ollama and LM Studio keep everything on your infrastructure
3. **Set up multiple providers**: Configure fallback options if your primary provider has issues
4. **Monitor token usage**: Track costs across different providers
5. **Provide clear context**: Give the AI specific information about what you're trying to accomplish

## Troubleshooting

### API Key errors

Verify your API key is correct and has appropriate permissions:
- Check for typos in the key
- Ensure the key hasn't expired
- Verify billing is set up for the provider

### Connection failures

For local providers (Ollama, LM Studio):
- Ensure the service is running
- Check the URL is correct
- Verify firewall settings allow local connections

### Rate limiting

If you hit rate limits:
- Add delays between requests
- Switch to a different provider
- Upgrade your API plan

### Poor response quality

Improve AI responses by:
- Using more capable models
- Providing more context in your questions
- Being specific about what you need
